## 1. Новый инструмент `compare_periods`

Добавлен инструмент:

```python
def compare_periods(metric: str, period_a: str, period_b: str) -> dict:
    """
    metric: "key_rate" | "fx_USD" | "fx_EUR" | "fx_CNY" | "cpi" | "unemployment"
    period_a/period_b: "YYYY-MM" или "YYYY-MM-DD"
    """
```

Инструмент:

- сравнивает одну метрику в двух периодах;
- возвращает `a`, `b`, `delta`, `ratio`, `source`;
- использует существующие функции `get_fx_rate`, `get_key_rate`, `get_inflation`, `get_unemployment` (логика не переписана с нуля);
- описан в `starter/schemas.py`.

Примеры запросов, где вызывается `compare_periods`:

- `Во сколько раз вырос курс USD с января 2022 по апрель 2026?`
- `Сравни ключевую ставку в 2022-02 и в 2026-04: на сколько процентных пункта изменилась?`
- `Сравни инфляцию за 2024-02 и 2024-03 ...`

## 3. Диагностика (3 разных типа ошибок)

Диагностика выполнена по `starter/trace.jsonl`.

### Тип ошибки 1: инструмент упал на отсутствии данных за месяц

Трасса:

```json
{"step": 1, "call": "get_inflation", "args": {"year": 2026, "month": 5}, "obs": {"error": "нет данных ИПЦ на 2026-05"}}
{"step": 2, "call": "get_inflation", "args": {"year": 2026, "month": 4}, "obs": {"error": "нет данных ИПЦ на 2026-04"}}
{"step": 3, "call": "get_inflation", "args": {"year": 2026, "month": 3}, "obs": {"year": 2026, "month": 3, "cpi_yoy": 6.8, "source": "rosstat_csv"}}
```

Причина:

- у месячных рядов Росстата есть лаг публикации, агент сначала пробует более свежие месяцы, которых ещё нет.

Что бы исправил:

- добавить в инструмент режим `latest_available`, чтобы сразу брать последний доступный месяц без нескольких неудачных вызовов.

### Тип ошибки 2: неправильный выбор инструмента

Трасса:

```json
{"step": 1, "call": "get_inflation", "args": {"year": 2024, "month": 2}, "obs": {"year": 2024, "month": 2, "cpi_yoy": 7.69}}
{"step": 1, "call": "get_unemployment", "args": {"year": 2024, "month": 3}, "obs": {"year": 2024, "month": 3, "unemployment": 2.7}}
{"step": 2, "call": "calculate", "args": {"expression": "7.72 / 7.69"}, "obs": {"result": 1.003901}}
```

Причина:

- на задаче сравнения периодов агент не выбрал специализированный `compare_periods`, а пошёл ручной цепочкой.

Что бы исправил:

- усилить правило в системном промпте: при формулировках "сравни между A и B" приоритетно вызывать `compare_periods`.

### Тип ошибки 3: неправильный порядок/полнота вызовов (пропуск `calculate`)

Трасса:

```json
{"step": 1, "call": "get_key_rate", "args": {}, "obs": {"rate": 16.0}}
{"step": 2, "final": "По правилу 72: 72 / 16 = 4.5 года ..."}
```

Причина:

- агент выполнил арифметику в тексте финального ответа, хотя по policy должен считать через инструмент `calculate`.

Что бы исправил:

- добавить проверку перед финалом: если в ответе есть вычисления, но в трассе нет `calculate`, отправлять агента на пересчёт.

## 4. Вывод: самый хрупкий элемент системы

Самое хрупкое место — **оркестрация на уровне LLM (tool-selection policy)**, а не сами функции инструментов.

Почему:

- инструменты в целом стабильны и предсказуемо возвращают либо валидные данные, либо `error`;
- большинство провалов связано не с падением функций, а с выбором стратегии агентом:
  - не тот инструмент;
  - лишние ретраи по датам;
  - арифметика в тексте вместо `calculate`.
