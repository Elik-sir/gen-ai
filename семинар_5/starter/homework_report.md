## 1) Новый инструмент `compare_periods`

Сигнатура:

```python
def compare_periods(metric: str, period_a: str, period_b: str) -> dict:
    ...
```

Поддерживаемые метрики:
- `key_rate`
- `fx_USD`, `fx_EUR`, `fx_CNY`
- `cpi`
- `unemployment`

Что делает:
- Берёт значение одной метрики в двух периодах.
- Возвращает структуру `a`, `b`, `delta`, `ratio`, `source`.
- Внутри переиспользует существующие инструменты:
  - `get_fx_rate`
  - `get_key_rate`
  - `get_inflation`
  - `get_unemployment`

Примеры запросов, где вызван `compare_periods`:
1. `Во сколько раз вырос курс USD с января 2022 по апрель 2026?`
2. `Сравни ключевую ставку в 2022-02 и в 2026-04: на сколько процентных пункта изменилась?`
3. `Сравни инфляцию за 2024-02 и 2024-03 и безработицу за те же месяцы: где изменение сильнее в относительных терминах?`

## 3) Диагностика: 3 разных типа ошибок

### Ошибка типа A: инструмент вернул `error` (нет данных на последний месяц)

Трасса:

```json
{"run_id":"95c618e3-...","step":1,"call":"get_inflation","args":{"year":2026,"month":5},"obs":{"error":"нет данных ИПЦ на 2026-05"}}
{"run_id":"95c618e3-...","step":2,"call":"get_inflation","args":{"year":2026,"month":4},"obs":{"error":"нет данных ИПЦ на 2026-04"}}
{"run_id":"95c618e3-...","step":3,"call":"get_inflation","args":{"year":2026,"month":3},"obs":{"year":2026,"month":3,"cpi_yoy":6.8,"source":"rosstat_csv"}}
```

Причина:
- В данных Росстата есть лаг публикации, а агент сначала пытается текущие месяцы.

Что исправить:
- Добавить отдельный инструмент/режим `latest_available=true` для месячных рядов, чтобы сразу брать последний доступный месяц без повторных попыток.

### Ошибка типа B: неправильный выбор инструмента (игнор `compare_periods`)

Трасса:

```json
{"run_id":"a4d39bbe-...","step":1,"call":"get_inflation","args":{"year":2024,"month":2},"obs":{"year":2024,"month":2,"cpi_yoy":7.69,"source":"rosstat_csv"}}
{"run_id":"a4d39bbe-...","step":1,"call":"get_unemployment","args":{"year":2024,"month":3},"obs":{"year":2024,"month":3,"unemployment":2.7,"source":"rosstat_csv"}}
{"run_id":"a4d39bbe-...","step":2,"call":"calculate","args":{"expression":"7.72 / 7.69"},"obs":{"expression":"7.72 / 7.69","result":1.003901}}
```

Причина:
- Вопрос подразумевает сравнение периодов, но агент выбирает ручную декомпозицию вместо специализированного инструмента.

Что исправить:
- Ужесточить системное правило: при формулировках `сравни X между A и B` вызывать `compare_periods` приоритетно.
- Добавить несколько демонстрационных примеров в prompt (few-shot) именно на такие формулировки.

### Ошибка типа C: пропуск обязательной арифметики через `calculate`

Трасса:

```json
{"run_id":"7d82a6cf-...","step":1,"call":"get_key_rate","args":{},"obs":{"rate":16.0,"date":"2026-06-14","valid_from":"2026-03-20","source":"fallback_csv"}}
{"run_id":"7d82a6cf-...","step":2,"final":"... По правилу 72: 72 / 16 = 4.5 года ..."}
```

Причина:
- Несмотря на правило в системном промпте, агент иногда считает простые выражения «в голове».

Что исправить:
- Добавить post-check: если в финальном ответе есть арифметическое выражение (`/`, `*`, `+`, `-`) без вызова `calculate` в trace, возвращать агента на пересчёт.

## 4) Вывод о хрупкости системы

Самый хрупкий элемент — **policy-level orchestration** (то, как LLM выбирает инструменты и порядок вызовов), а не сами функции инструментов.

Почему:
- Инструменты в целом работают стабильно и возвращают данные/ошибки предсказуемо.
- Основные сбои возникают из-за поведения агента:
  - не тот инструмент,
  - лишние ретраи по датам,
  - ручная арифметика вместо `calculate`.

Практический приоритет улучшений:
1. Жёстче policy выбора инструментов (`compare_periods` по шаблонам вопроса).
2. Автоматические guardrails для арифметики и проверки цепочки вызовов.
3. Умный helper для последних доступных месячных данных.
