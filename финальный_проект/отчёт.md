# Отчёт — финальный проект (Трек B)

**Тема:** Q&A-ассистент по корпусу статей Habr про RAG и LLM-агентов.

**Запуск:** `python scripts/fetch_habr.py && python pipeline.py --ingest && python eval.py`

---

## 1. Задача и данные

### Задача

Построить прикладной конвейер, который отвечает на вопросы по корпусу технических статей, опираясь только на retrieved-контекст, возвращает структурированный ответ с цитатами и chunk_id, проверяет галлюцинации и оценивается на gold-наборе из 18 кейсов.

Предметная область сознательно совпадает с темой курса (RAG, агенты, eval), но **данные не из семинаров** — это свежий корпус публичных статей Habr, собранный скриптом `scripts/fetch_habr.py`.

### Корпус

| Параметр            | Значение                                             |
| ------------------- | ---------------------------------------------------- |
| Документов          | 10                                                   |
| Символов            | 273 840                                              |
| Чанков после ingest | 991                                                  |
| Chunking            | RecursiveCharacterTextSplitter, size=400, overlap=80 |
| Индекс              | ChromaDB (dense) + BM25 cache + RRF                  |

**Источники** (полный список — `input/corpus_meta.json`):

| ID                       | Статья                            |
| ------------------------ | --------------------------------- |
| `habr_rag_deep_dive`     | RAG: глубокий технический обзор   |
| `habr_rag_intro`         | Что такое RAG                     |
| `habr_rag_simple`        | RAG — простое объяснение          |
| `habr_rag_smart_search`  | RAG или умный поиск по документам |
| `habr_rag_python`        | Создаём RAG на Python (OTUS)      |
| `habr_react_chestnyznak` | От Naive RAG до ReAct-агента      |

| `habr_react_sber`        | ReAct-агент (LangGraph)           |

**Подготовка:** HTML статей парсится через BeautifulSoup (`div.tm-article-body`), сохраняется как `.txt` с заголовком и URL. Повторная загрузка — идемпотентна.

### Gold-набор (`input/gold.json`)

18 вопросов: 12 in-scope (прямой / перефраз / multi-hop), 2 out-of-scope, 1 adversarial. Для каждого указаны `gold_sources`, эталонный ответ и флаг `expect_abstain`.

## 2. Система

### Техники курса (5)

| #   | Техника                             | Модуль             | Зачем                                                                 |
| --- | ----------------------------------- | ------------------ | --------------------------------------------------------------------- |
| 1   | **Hybrid RAG** (BM25 + dense + RRF) | `rag.py`           | Находить релевантные чанки по синонимам и ключевым словам             |
| 2   | **Агент с инструментами** (ReAct)   | `agent.py`         | Модель сама решает, когда и как искать; несколько search на multi-hop |
| 3   | **Структурированный вывод**         | `schema.py`        | `RAGAnswer` с `max_retries`; бизнес-инвариант status↔confidence       |
| 4   | **LLM-as-judge**                    | `judge.py`         | Независимая проверка: ответ следует из chunks или это галлюцинация    |
| 5   | **Проверка галлюцинаций**           | `hallucination.py` | Ghost-цитаты, fake chunk_id, unsupported numbers                      |

### field_validator

В `schema.py` — связка `status` и `confidence`:

- `insufficient` → confidence ≤ 0.35, quotes/chunk_ids могут быть пустыми
- `answered` → confidence ≥ 0.45, обязательны quotes и chunk_ids

### Схема пути данных

```
Вопрос пользователя
  → ReAct-агент (agent.py)
      → tool: search_corpus → hybrid RAG (rag.py) → top-K чанков
      → tool: submit_answer → RAGAnswer (schema.py)
  → ghost-check (hallucination.py)
  → LLM-as-judge (judge.py)
  → output/run_answers/*.json + trace.jsonl
```

### Инструменты агента

- `search_corpus(query, k)` — гибридный retrieval, возвращает chunk_id + text
- `submit_answer(...)` — финальный structured output

Системный промпт запрещает отвечать без search и требует `insufficient`, если контекста нет.

## 3. Оценка

**Команда:** `python eval.py` (полный прогон, 18 кейсов, ~7.7 мин).

### Критерий pass

```
pass = correctness AND total_hallucination_issues == 0
```

Для in-scope: judge ∈ {supported, partially_supported} + hit-rate@5 ≥ 0.5 + status=answered.  
Для out-of-scope: status=insufficient + judge=abstain_ok.

## 4. Где пайплайн ошибался

### Провал 1 — Q3: этапы RAG-пайплайна (retrieval + ghost)

**Вход:** «Какие этапы включает стандартный RAG-пайплайн?»

**Retrieval:** hit-rate **0.0**. В top-5 попали только `habr_hybrid_rag` и `habr_rag_deep_dive`; нужные `habr_rag_smart_search` и `habr_rag_python` — нет.

**Ответ агента:** содержательно верный (judge=supported, score=1.0), но **1 ghost-цитата** — модель слегка переформулировала фразу из чанка.

**Почему:** общая формулировка вопроса; dense-поиск тянет «обзорные» статьи, а пошаговое описание pipeline — в других документах. Агент компенсировал тремя search, но ghost-gate всё равно завалил pass.

---

### Провал 2 — Q18: adversarial (блокчейн)

**Вход:** «Использует ли RAG блокчейн для хранения embeddings?»

**Поведение:** агент корректно вернул `status=insufficient`, confidence=0.1, judge=abstain_ok — **содержательно правильный отказ**.

**Искажение:** в поле `quotes` осталась **ghost-цитата** (модель прикрепила фрагмент, которого нет в retrieved chunks). Pass=false из-за hallucination gate, хотя abstain сработал.

**Урок:** при `insufficient` нужно принудительно обнулять `quotes` в post-processing или в validator.

---

### Провал 3 — Q13: multi-hop (ReAct + RAG)

**Вход:** «Как ReAct-агент использует RAG-поиск и чем это лучше статичного пайплайна?»

**Путь:** 6 шагов, **4 search**, 24 468 prompt tokens (~$0.004), latency 51.7 сек.

**Retrieval:** hit-rate 0.5 — нашли `habr_react_chestnyznak`, но не `habr_agent_prod`.

**Ответ:** judge=supported, но 1 ghost-цитата. Агент дважды вызывал `submit_answer` (validation retry на первой попытке).

**Почему:** multi-hop требует информацию из двух статей; без reranker второй источник не всегда попадает в контекст, а модель «подтягивает» цитату пересказом.

## 5. Выводы

### Ответ на задачу

Система **работает как Q&A по корпусу статей из Habr**: на 15 in-scope вопросах judge дал `supported` (100%), на 3 out-of-scope/adversarial — корректный `abstain_ok`. End-to-end pass-rate 50% — следствие **строгого ghost-gate**, а не массовых содержательных ошибок.

Retrieval-only (77.8%) vs end-to-end (50%) показывает: узкое место — не генерация, а **качество цитирования** и иногда retrieval на общих вопросах.

### Что оставил бы в прод

- Hybrid RAG (BM25 + dense + RRF) — стабильный baseline без фреймворков
- Structured output с `field_validator` (status↔confidence)
- Ghost-quote checker как blocking gate перед отдачей пользователю
- ReAct-агент с `search_corpus` — для multi-hop и перефразов запроса
- Trace + token accounting в `output/trace.jsonl`

### Что переделал бы

1. **Post-process при insufficient:** очищать `quotes` и `chunk_ids`
2. **Fuzzy match для ghost-check:** нормализация пунктуации / порог Levenshtein вместо substring[:30]
3. **Reranker** (cross-encoder) после RRF — особенно для multi-hop
4. **Judge не на каждый запрос** — только на sample или при ghost>0 (экономия ~1 LLM-call)
5. **Один submit_answer** — жёстче обрабатывать validation error без повторного tool-call

### Границы применимости

- Корпус статичен: новые статьи Habr не подхватятся без re-ingest
- Язык и домен: multilingual MiniLM + русскоязычные статьи; перенос на другой домен требует нового eval
- Out-of-domain: abstain работает (2/2 чистых pass), но adversarial с ghost-цитатой при отказе — слабое место
- Multi-hop дорог по токенам (Q13: 24k prompt tokens) — без бюджета и reranker не масштабируется
- Judge и генератор — одна модель (монокультура); для продакшена разделил бы роли
